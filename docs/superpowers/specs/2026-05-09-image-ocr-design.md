# RedBook Fixer 图片 OCR 最小闭环设计

## 1. 背景

当前项目已经支持“小红书文案敏感词检测与替换”，但用户在真实使用中还存在一类高频风险：图片中的文字没有经过敏感词处理，导致内容被限流。

本次目标是在现有纯前端静态站基础上，新增一个独立的“图片识别文字”流程，用于从单张图片中提取文字，再将提取结果送入现有“文案敏感词检测”流程。

本次设计遵循两个明确约束：

1. 图片识别文字和文案敏感词检测需要分开。
2. 第一版优先跑通单张图片的最小闭环，不追求批量能力。

## 2. 目标

本次功能需要实现以下结果：

1. 用户可以通过点击上传、拖拽上传、直接粘贴截图三种方式导入单张图片。
2. 页面内可以展示当前图片预览。
3. 用户可以在前端本地完成图片文字识别，不依赖后端服务。
4. 识别结果以纯文本形式展示，允许用户手动编辑修正。
5. 用户可以一键将当前识别文本送到“文案处理”页面。
6. 送检动作只做“自动填充并切换页面”，不自动触发敏感词检测。

## 3. 非目标

以下内容不在本次范围内：

1. 批量图片上传与批量 OCR。
2. 多张图片识别结果汇总。
3. 在原图上标记敏感词位置。
4. 图片中文字框选修正。
5. OCR 结果自动持久化保存。
6. 移动端专项适配。
7. 第三方 OCR API 接入。

## 4. 现有项目上下文

当前项目是一个纯静态单页应用：

1. 主要 UI、样式、业务逻辑全部集中在 `index.html`。
2. 内置敏感词词库来自 `words.json`。
3. 自定义词库存储于浏览器 `localStorage`。
4. 当前页面结构为 `文案处理 / 词库管理 / 关于作者` 三个 Tab。

基于当前结构，第一版应优先采取最小增量方案，在现有单页内新增 `图片识别` Tab 和对应逻辑，而不是先做大规模拆分。

## 5. 方案选择

评估过的方案如下：

### 方案 A：最小增量方案

在当前单页结构中新增 `图片识别` Tab，并将 OCR 相关 UI 与逻辑直接加入现有 `index.html`。

优点：

1. 改动路径最短，最适合先跑通闭环。
2. 与现有项目的单文件结构一致。
3. 不需要引入额外打包流程或工程化改造。

缺点：

1. `index.html` 会继续变大。
2. 后续扩展批量功能时，维护成本会上升。

### 方案 B：小幅拆层方案

保持静态站不变，但把 OCR 逻辑与文案处理逻辑拆分为独立 JS 模块。

优点：

1. 可维护性更好。
2. 后续扩展批量处理更自然。

缺点：

1. 本次改动面更大。
2. 不符合“最小闭环优先”的原则。

### 方案 C：独立图片工具子页

把图片 OCR 做成相对独立的子流程或子页面，再与文案检测页做跳转衔接。

优点：

1. 隔离性更强。

缺点：

1. 状态传递和页面衔接复杂度更高。
2. 相比当前项目规模，收益不明显。

### 结论

本次采用方案 A。

## 6. 页面与交互设计

### 6.1 顶部导航

顶部导航调整为四个 Tab：

1. 文案处理
2. 图片识别
3. 词库管理
4. 关于作者

### 6.2 图片识别页面结构

新增 `图片识别` 页面，包含四个核心区域：

1. 图片导入区
2. 图片预览区
3. OCR 结果区
4. 操作区

### 6.3 图片导入区

支持三种输入方式：

1. 点击上传本地图片
2. 拖拽图片到指定区域
3. 直接粘贴截图

导入区职责：

1. 提示支持的三种导入方式。
2. 校验文件是否为图片。
3. 接收并设置当前图片。

### 6.4 图片预览区

在页面中展示当前导入的单张图片，用于确认识别目标是否正确。

第一版不支持：

1. 缩放编辑
2. 裁剪
3. 多图切换

### 6.5 OCR 结果区

OCR 完成后，以纯文本形式展示识别结果，并允许用户直接编辑。

该区域需要支持：

1. 未识别前显示占位提示。
2. 识别后填充文本结果。
3. 用户手动修改结果。

该区域不负责：

1. 敏感词高亮
2. 替换建议
3. 图片原图标注

### 6.6 操作区

至少包含以下按钮：

1. `开始识别`
2. `清空`
3. `一键送去检测`

交互规则：

1. 未导入图片时，`开始识别` 不可执行。
2. OCR 识别中时，禁止重复点击 `开始识别`。
3. OCR 结果为空时，`一键送去检测` 不可执行。
4. `清空` 会清除当前图片、识别结果、错误提示。

## 7. OCR 技术方案

本次 OCR 采用纯前端本地识别方案，首选 `Tesseract.js`。

原因如下：

1. 与当前纯静态站形态一致。
2. 不需要后端服务。
3. 不需要将图片发送到外部接口。
4. 更符合用户对现有工具“本地使用、低外部依赖”的预期。

第一版识别重点：

1. 中文为主
2. 顺带支持英文和数字

## 8. 数据流设计

### 8.1 页面内状态

图片识别页需要维护独立状态：

1. `currentImageFile`：当前图片文件
2. `currentImageUrl`：预览用对象 URL
3. `ocrText`：当前 OCR 结果文本
4. `ocrLoading`：识别中状态
5. `ocrError`：错误提示信息

文案处理页继续沿用当前状态：

1. 输入文本
2. 检测结果
3. 替换结果

### 8.2 页面间衔接

页面之间只共享“送检文本”这一条通道。

流程如下：

1. 用户导入单张图片。
2. 页面展示图片预览。
3. 用户点击 `开始识别`。
4. 前端本地执行 OCR。
5. OCR 结果写入图片页文本区。
6. 用户可手动编辑结果。
7. 用户点击 `一键送去检测`。
8. 页面切换到 `文案处理` Tab。
9. 当前 OCR 文本自动填入文案输入框。
10. 不自动开始敏感词检测，等待用户手动点击。

## 9. 异常处理

### 9.1 输入异常

1. 未上传图片就点击 `开始识别`：提示用户先导入图片。
2. 上传非图片文件：直接拦截并提示。
3. 粘贴内容不包含图片：提示未检测到图片内容。

### 9.2 OCR 过程异常

1. 识别过程中重复点击：按钮禁用并显示处理中状态。
2. OCR 执行失败：保留图片预览并提示重试或更换清晰图片。
3. OCR 结果为空：允许用户手动输入，但提示未识别到有效文字。

### 9.3 送检异常

1. 识别结果为空时点击 `一键送去检测`：不切换页面，直接提示。

## 10. 第一版实现边界

为了保证本次工作聚焦，以下原则必须成立：

1. 不引入后端。
2. 不引入复杂工程化改造。
3. 不为批量场景预埋过度抽象。
4. 不在图片页复用敏感词检测 UI。
5. 只做“识别文字 -> 编辑 -> 送去文案检测”的闭环。

## 11. 验收标准

满足以下条件即可认为第一版完成：

1. 用户可通过上传、拖拽、粘贴三种方式导入单张图片。
2. 图片导入后可正常预览。
3. 点击 `开始识别` 后可以得到可编辑文字结果。
4. 识别过程具备基本状态反馈与错误提示。
5. 点击 `一键送去检测` 后，能切换到 `文案处理` 页并自动填入文本。
6. 文案页的现有敏感词检测流程不被破坏。
7. 整个流程在本地静态站内可运行，不依赖任何后端接口。

## 12. 后续演进方向

本次完成后，下一阶段优先考虑：

1. 批量图片上传。
2. 多图识别结果汇总后统一送检。
3. 结合批量场景对单文件逻辑做适度拆分。
